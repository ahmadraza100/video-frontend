VidShare — Short video frontend (React + Vite + TypeScript)

What this app does
------------------
VidShare is a lightweight short-video sharing frontend. It provides a modern UI for creators and viewers to upload, discover, watch, like, comment on, bookmark, and manage short videos.

Key features
------------
- Upload videos with generated thumbnails (upload flow in `/src/pages/Upload.tsx`).
- Video playback with an in-app player that supports poster images, play/pause, mute and seeking (`/src/components/video/VideoPlayer.tsx`).
- Video listing and discovery pages (Home, Explore) with category filters.
- User profiles with video lists and simple profile editing (`/src/pages/Profile.tsx`).
- Video view page with comments and interaction controls (like/bookmark/share). Comments endpoints and hooks available (`/src/hooks/useInteractions.ts`).
- API client abstraction in `/src/lib/api.ts` and a media URL helper in `/src/lib/utils.ts` that converts API-provided relative paths (e.g. `/uploads/...`) into absolute URLs using `VITE_API_URL`.

Developer notes
---------------
- Media paths: the backend often returns relative media paths (e.g. `/uploads/videos/...`). The helper `buildMediaUrl()` in `/src/lib/utils.ts` ensures these are converted to usable absolute URLs using the `VITE_API_URL` environment variable. If `videoUrl` or `thumbnail` is already absolute (http/https) or a `blob:`/`data:` URL, the helper returns it unchanged.
- API base URL: controlled via `import.meta.env.VITE_API_URL`. Default fallback used in code: `https://video-backend-769.azurewebsites.net`.
- Comments: there is a `commentAPI` (new) in `/src/lib/api.ts` that exposes paginated listing and create/update/delete/like comment endpoints. Use `useInteractions` and/or call `commentAPI` directly for server-side pagination and sorting.
- Deleting videos: frontend hooks include a delete flow (`useVideos.deleteVideo`) and the UI has delete options (profile list and video view). When deleting, the UI confirms and then navigates to the profile/home page on success.

Production & deploy notes (Azure Static Web Apps)
-------------------------------------------------
- Azure Static Web Apps does not use .htaccess. For SPA route handling add `staticwebapp.config.json` to the published artifact root. This repository already contains `public/staticwebapp.config.json` which rewrites unknown routes to `/index.html` and excludes API and static assets.
- Ensure Vite copies `public/` into the build output (default behaviour). Deploy the `dist/`/`build` artifact generated by your CI to Azure Static Web Apps.

Deployment via GitHub Actions (Azure Static Web Apps)
----------------------------------------------------
- This project can be deployed to Azure Static Web Apps using the official GitHub Actions workflow. Typical setup:
	- Create an Azure Static Web App in the Azure portal and connect it to the GitHub repository. Azure will create or suggest a GitHub Actions workflow file at `.github/workflows/azure-static-web-apps-<env>.yml`.
	- The workflow builds the Vite app (usually `npm run build`) and deploys the `dist/` output to Azure. It also wires the `staticwebapp.config.json` file so client-side routing works on reload.
	- Make sure `VITE_API_URL` is set in Static Web Apps environment configuration (Application settings) to point to your backend (for production use the production backend URL, not the `/api` path).

Example notes for the workflow:

	- Build step: `npm run build` (or `yarn build` / `bun build`).
	- App artifact location: `dist/` (ensure the Action's `app_location`/`output_location` settings match your Vite output).
	- Environment variables: set `VITE_API_URL` in the workflow or (preferably) in the Static Web App configuration in Azure so secrets and endpoints are not in the repo.

If you already have a workflow created by Azure when you provisioned the Static Web App, GitHub Actions will run on push to the configured branch and automatically deploy the built site.
